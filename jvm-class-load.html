加载

加载过程就是将 Class 文件加载到 JVM 内存（方法区）中并生成 Class 对象的过程

class 是类加载器通过双亲委派来加载的

双亲委派模型

所谓的双亲委派就是 当一个 ClassLoader loadClass 的时候先去缓存里找，

如果缓存命中直接返回，否则让父加载器加载

父加载器也是先去缓存里找，如果缓存命中直接返回，否则让父加载器的父加载器加载

直到父加载器为 null 也就是最顶层的父加载器 BootstrapClassLoader 时还没找到

BootstrapClassLoader 首先去自己负责加载的 jar 或 class 中查找，

如果找到加载并返回，否则委托给 子加载器也就是 ExtClassLoader 加载

ExtClassLoader 首先去自己负责加载的 jar 或 class 中查找，，

如果找到加载并返回，否则委托给 子加载器也就是 AppClassLoader 加载

当子加载器是这个调用 loadClass 方法的加载器时，去自己负责加载的 jar 或 class 中查找，

如果找到这个类加载并返回，否则抛出 ClassNotFoundException

小结：先查找在加载

为什么双亲委派

1、主要为了安全

2、防止加载重复的类

类加载器

类加载器分为 BootstarpClassloader、ExtClassloader、AppClassloader还有自定义类加载器

BootstarpClassloader 负责加载下面这些 jar 包和 classes 或者通过 -Dsun.boot.class.path

D:\java\jdk\jdk1.8.0_221\jre\lib\jsse.jar

ExtClassloader 负责加载下面这些 jar 包和 classes 

D:\java\jdk\jdk1.8.0_221\jre\lib\ext
C:\windows\Sun\Java\lib\ext

AppClassLoader 负责 classpath 下面的 jar 包

问题来了, AppClassloader 和 ExtAppClassLoader 是谁加载的？

ExtClassloader、AppClassloader 是 BootstarpClassloader 加载的

另外自定义类加载器的是 AppClassloader 加载的

自定义类加载器

重写 findClass 方法

@Override
protected Class<?> findClass(String name) throws ClassNotFoundException {
    try(InputStream is = new FileInputStream("D:/test/" + name.replaceAll("\\.", "/") + ".class")) {
        byte[] bytes = new byte[is.available()];
        is.read(bytes);
        return defineClass(name, bytes, 0, bytes.length);
    } catch (FileNotFoundException e) {
        throw new ClassNotFoundException(name);
    } catch (IOException e) {
        e.printStackTrace();
    }
    return super.findClass(name);
}

执行模式

解释模式

通过解释器（Bytecode Interpreter）解释执行 特点：启动快（不需要编译），执行慢 可通过-Xint参数指定为纯解释模式

编译模式

由JIT（Just In-Time Compiler）编译为本地代码（C语言实现）执行 特点：启动慢（编译过程较慢），执行快 可通过-Xcomp参数指定为纯编译模式

混合模式

混合使用解释器 + 热点代码编译 起始阶段采用解释执行 热点代码检测（HotSpot），默认-XX:CompileThreshold=10000 多次被调用的方法（方法计数器：监测方法执行频率） 多次被调用的循环（循环计数器：监测循环执行频率） 对热点代码进行编译 默认采用这种模式，可通过-Xmixed指定
